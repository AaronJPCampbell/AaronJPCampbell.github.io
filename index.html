Plan

As mentioned above, I would also like to expand the data stored for each data to include calorie data.

The Algorithms and Data Structure will demonstrate that I am capable of producing and utilizing algorithms based on entered data, and am able to use this to enhance my program. Finally, the Database features/enhancements will illustrate the best practices I learned during my database-related course(s). The planned enhancements are all within my capabilities, as they are all based on skills and practices I have learned throughout this program. I feel that my only concern is regarding the UI enhancements. In my Mobile Architecture and Programming course, the UI was important, but it was more important to get the code working as intended, so I did not spend as much time looking into UI enhancements for Android Studio. I plan to look into these for this project, but this particular portion might be rather difficult for me to address. Algorithms and Data Structure Enhancements

I had an extremely tough time getting these enhancements working. On my first attempt, nothing really worked as I wanted it to. I had the code for the weight data line graph implemented (consisting of code initializing the graph, a method to pull specific weight entries to display, and the code to add these weight entries to the graph), but the graph would not show up at all in my app; and the algorithm to calculate expected weight loss was entered into my code, but did not seem to actually work. Adding to my frustratons was the fact that any minor edit to the aforementioned bits of code almost always resulted in the app crashing on login. After my initial submission, I spent a good amount of time combing through my code, and finally found the sources of all the errors I was getting. Once I adjusted this so it was correct, I was able to successfully pull weight and calorie data from their respective databases based on a specified date. This allowed me to not only move forward creating my weight data line graph, but also allowed me to complete the calorie budget and projected weight loss algorithms (this allowed me to finalize the Calorie Budget item from the previous enhancement). Finally, just as with the previous enhancement, I realized my code was ordered incorrectly, and by adjusting that I was able to get everything displayed on the app without any crashes.

I also made some additional UI enhancements, all very minor (e.g. text size changes) save for one: I added a scrolling effect to the main page of the app. In the previous enhancement, I realized I was going to run out of room on the page, so I looked into adding the ability to scroll the page. Thankfully this was a very simple change to make, as all it consisted of was nesting my XML code under a ScrollView layout.

Database Enhancements

For the Database Enhacements, my initial plan was to find a way to normalize data entry into the weight database. My un-enahnced app allowed dates to be entered in whatever manner the user wanted (2/10/23, 2/10/2023, Feb. 10 2023, etc.), which needless to say would cause issues with the data. I also wanted to create another database to store calorie data, as well as create a function that could look up data based on a specified date (outside of the update and delete functions already present). As mentioned previously, the calorie database and its related functions had already been implemented at this point, leaving me to work mainly on date normalization. I was able to implement this by doing the following:

  • Setting up a LocalDate variable that was set to the current date in YYYY-MM-DD format
  • Altering the code so that, rather than specifying the date when entering new data, the date is set to today’s date instead via the LocalDate variable

These changes ensured that all data entered into the database had date entries in YYYY-MM-DD format. This also ensured that data would be entered chronologically, allowing me to employ functions that required the data be in date order (such as the line graph). In addition to this, the code was previously set up so that any of the updates or deletions of database entries would be rejected if no matching date was found, so the user would need to normalize dates on their end in order to perform operations such as these. I did face an issue where the LocalDate variable did not behave well with searching the databases, since the dates in them were stored as Strings. I found the simplest way to fix this was to have another variable which converted today’s date to a String and use that new String variable as the main one. I also did the same for the 6 days prior to today, so that the line graph could be displayed.

Finally, regarding the data lookup based on date, I had started work on this for the previous enhancement (it was necessary for some of the algorithms and the line graph to function), but was initially unable to get it working as intended. After spending a good amount of time trying to figure out what exactly was wrong, I realized that it was simply an incorrect query. In the weight database lookup method, I initially put:

  • Cursor cursor = db.rawQuery("Select weight from WeightDetails where date = ?", new String[] {date});
No matter what I did it seemed this method was causing my app to crash. Then I decided to make a minor edit:
  • Cursor cursor = db.rawQuery("Select * from WeightDetails where date = ?", new String[] {date});
This edit fixed the issues I was experiencing, and allowed me to create the lookup method for the calorie database.
